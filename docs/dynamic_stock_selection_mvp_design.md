# Dynamic Stock Selection MVP Design

## Overview
This document defines the blueprint for a "Dynamic Stock Selection MVP." Following the failure of the static dictionary approach, this design relies purely on live financial data filtering combined with previously generated macro constraints. No LLMs or AI engines are introduced.

---

## TASK 1: Minimum Live Data Required

To successfully filter a broad sector into a targeted Akshat-style watchlist, the absolute minimum live data points required per ticker are:

1. **Current Price vs. 200-Day Moving Average (200DMA):** Measures entry velocity and parabolic risk.
2. **Trailing 12 Month (TTM) Operating Margin Trend:** Measures moat durability.
3. **Forward Price-to-Earnings (PE):** Measures current valuation.
4. **5-Year Average PE:** Provides the baseline for historical valuation gravity.
5. **Projected 3-Year EPS Growth Rate:** Measures earnings momentum.
6. **Debt-to-Equity Ratio:** Measures balance sheet fragility.

---

## TASK 2: Review of Akshat's Stock-Selection Metrics

Based on the core philosophy established in `01_Akshat_Master_System.md`, the metrics are triaged as follows:

**Mandatory:**
*   *Forward PE vs Historical PE* (Rule: Earnings gravity).
*   *Operating Margin Trend* (Rule: K-shaped/Profit expansion).
*   *Price vs 200DMA* (Rule: Never buy parabolic setups).

**Useful:**
*   *Debt-to-Equity* (Rule: Survive W-shaped structural damage).
*   *EPS Growth vs Price CAGR* (Rule: Ensure price hasn't outrun earnings).
*   *ROCE* (Return on Capital Employed).

**Optional:**
*   *Promoter Holding* (Helpful context, but not a strict disqualifier in all macro regimes).
*   *Order Book Size* (Highly sector-specific; mandatory for Defense/Infra, useless for FMCG).

---

## TASK 3: Dynamic Stock Selection Checklist

When a Sector is identified by the Framework, every constituent stock in that sector must pass this boolean filter:

- [ ] **Valuation Check:** Is `Forward PE` <= `(5-Year Avg PE * 1.2)`? *(Reject extreme overvaluation).*
- [ ] **Momentum Check:** Is `Current Price` <= `(200DMA * 1.3)`? *(Reject parabolic breakouts).*
- [ ] **Profitability Check:** Is `TTM Operating Margin` >= `Previous Year Operating Margin`? *(Reject margin compression).*
- [ ] **Growth Check:** Is `Projected EPS Growth` > `0`? *(Reject earnings decay).*
- [ ] **Survival Check:** Is `Debt-to-Equity` < `2.0`? *(Reject bankruptcy risk).*

## TASK 4: Resolving the Failed Stock Tests via Live Data

*(Reviewing the hypothetical stock failures from previous red-team audits)*

**1. Semiconductor Incentives → Recommends Intel**
*   *Would live data prevent this?* **YES.** Intel's live TTM Operating Margins have frequently been negative or compressing year-over-year. The *Profitability Check* would instantly reject it, leaving only margin-expanding peers like TSMC or Nvidia.

**2. Hospital Expansion → Recommends Tata Motors**
*   *Would live data prevent this?* **YES.** While Tata Motors might occasionally pass margin checks, the *Valuation Check* or *Momentum Check* combined with the specific sub-sector filtering would isolate the true K-shaped beneficiaries. (Note: A static sub-sector map is still needed to separate Hospitals from Auto, but live data prevents buying *overvalued* or *unprofitable* Auto).

**3. Data Center Growth → Recommends Regional Utility**
*   *Would live data prevent this?* **YES.** The *Survival Check* (Debt-to-Equity < 2.0) would instantly flag the highly-leveraged utility company, rejecting it from the watchlist and protecting the portfolio from rate-hike vulnerability.

## TASK 5: Smallest Possible Implementation

**Sector → Candidate Stocks (Without AI)**

The smallest viable implementation is a **Python Boolean Pipeline**:

1.  **Input:** An array of Sector Strings (e.g., `["Semiconductors", "Power Generation"]`) generated by `2_collide.py`.
2.  **API Call:** Connect to a free financial API (e.g., Yahoo Finance via `yfinance` Python library). Query all tickers categorized under the input sectors.
3.  **Data Fetch:** Download the 6 mandatory live metrics for each ticker.
4.  **Filter Loop:** Run the data through the hardcoded *Dynamic Stock Selection Checklist*.
5.  **Output:** Return only the tickers that passed all boolean checks.

## FINAL VERDICT

**Can stock selection be solved using: Principles + Live Financial Data + Constraint Filtering, or is a deeper reasoning layer required?**

It can be solved using **Principles + Live Financial Data + Constraint Filtering.**

**Evidence:**
Akshat's stock selection logic is fundamentally quantitative, not qualitative. While Trend Discovery requires deep semantic reasoning to understand *how* a geopolitical event changes the world, the actual purchase of a stock is a math problem.

As stated in the Master System: *"Investing is primarily a mathematics-and-macro exercise... Price must eventually relate to earnings power."*

If the upstream AI/Routing engine correctly identifies the correct Sector (the macro), then simple deterministic constraints (the mathematics) are actually the *superior* way to select the stock. You do not need an LLM to reason about a P/E ratio; you just need a strict `if/else` statement ensuring you don't overpay. Once the sector is identified, live financial data and hard constraints perfectly replicate the defensive, valuation-sensitive philosophy of the system.
